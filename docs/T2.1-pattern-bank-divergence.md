# T2.1 — Pattern-bank "duplication" (M5): investigated, divergence documented

**Outcome: do NOT merge. The audit's "near-identical regex banks" premise does not hold;
a behavior-preserving single bank is not achievable. Documented here per the agreed
"verify zero detection change, else revert and document divergence" plan.**

## What the audit (M5) claimed
`behavioral-config.ts` and `capability-contract-config.ts` define "near-identical regex
banks with no shared import" (auth-state, credential-store persistence, browser-profile
copy), so a pattern update must land in two places.

## What's actually true (measured)
- **Whole pattern arrays byte-identical across the two files: 0.** Neither `behavioral`'s
  `BehavioralPattern.patterns[]` arrays (50) nor `capability`'s `*_PATTERNS` arrays (33)
  match each other as whole arrays.
- **Individual regex literals shared byte-for-byte: 42.** So there is real *literal-level*
  overlap, but the literals are **grouped differently**:
  - `behavioral-config.ts` uses **combined-alternation** regexes (`/(?:A|B|C)/i`) inside
    `{name, patterns, severity, deduction, owaspCategory, recommendation}` objects, because
    it emits *behavioral-risk findings* with per-pattern metadata.
  - `capability-contract-config.ts` uses **atomic** regexes in flat `readonly RegExp[]`
    arrays, because it does *capability inference* (`firstPositiveMatch` → first hit wins).

## Why a "single shared bank" is not behavior-preserving
The two analyzers consume patterns through different machinery:
- Merging behavioral's combined regexes into capability's atomic arrays (or vice-versa)
  changes what `firstPositiveMatch` and the behavioral matcher match, because alternation
  members are not 1:1 with the atomic patterns, and per-pattern negation / code-block /
  threat-listing context checks apply *per array element*. Splitting a combined regex into
  atomic elements (or fusing atomics into one alternation) changes the per-element context
  evaluation.
- A bounded experiment confirmed it: there is no array pair to swap without restructuring
  one analyzer's regex shape, which moves detection. The 308-test suite + the 224-sample
  detection benchmark (F1 0.8425) are the guardrails that would have caught — and would
  flag — any such change.

## Recommendation
Keep the two banks separate. They are **intentionally** different representations for two
different jobs (risk-finding emission vs capability inference), not a copy-paste to unify.
M5's real concern — *silent drift between the 42 shared literals* — is better addressed, if
it ever bites, by a small "shared-literal parity" guard test than by a structural merge
that trades a real regression risk for a cosmetic dedup.

(M2's other two refactors, T2.2 shared `score-util.ts` and T2.3 data-driven
`inferCapabilities`, were genuine safe dedups and landed — both behavior-preserving.)
